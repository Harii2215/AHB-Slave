# Contributing to AHB-Slave\n\nThank you for your interest in contributing to the AHB-Slave project! This document provides guidelines for contributing.\n\n## Code of Conduct\n\nBe respectful and professional in all interactions. All contributors are expected to maintain a positive and inclusive environment.\n\n## How to Contribute\n\n### 1. Reporting Bugs\n\nIf you find a bug, please create an issue with:\n- Clear title describing the problem\n- Detailed description of the issue\n- Steps to reproduce\n- Expected vs. actual behavior\n- Your simulation environment (tool, version, OS)\n\n### 2. Suggesting Enhancements\n\nEnhancement suggestions are welcome! Please create an issue with:\n- Clear title\n- Detailed description of the proposed feature\n- Rationale and use cases\n- Any related references or standards\n\n### 3. Pull Requests\n\n#### Before You Start\n1. Check existing issues and PRs to avoid duplication\n2. Fork the repository\n3. Create a feature branch: `git checkout -b feature/your-feature-name`\n\n#### Code Style Guidelines\n\n**Verilog/SystemVerilog:**\n- Use 4-space indentation\n- Use descriptive signal and variable names\n- Follow AMBA naming conventions (lowercase with underscores)\n- Add comments for complex logic\n- Use lowercase for keywords\n\nExample:\n```verilog\nalways @(posedge clk or negedge hresetn) begin\n    if (!hresetn) begin\n        state <= IDLE;\n    end else begin\n        case (state)\n            IDLE: begin\n                if (hsel && hready) begin\n                    state <= CHECK_MODE;\n                end\n            end\n            // ... rest of FSM\n        endcase\n    end\nend\n```\n\n**Documentation:**\n- Use clear, concise language\n- Include examples where helpful\n- Update README.md if adding features\n- Add comments to complex verification code\n\n#### Testing Requirements\n\nBefore submitting a PR:\n1. Test all modifications with your simulation tool\n2. Verify existing test cases still pass\n3. Add new test cases for new features\n4. Update documentation as needed\n5. Ensure no simulation errors or warnings\n\n#### Commit Message Guidelines\n\nUse clear, descriptive commit messages:\n```\nBrief description (50 chars or less)\n\nLonger explanation if needed (wrap at 72 chars)\n- Include what changed\n- Explain why the change was made\n- Reference any related issues (#123)\n```\n\nExamples:\n- ✅ `Fix address calculation for WRAP16 bursts`\n- ✅ `Add support for error response generation`\n- ❌ `Fixed bug`\n- ❌ `Update`\n\n#### Submission Checklist\n\n- [ ] Code follows style guidelines\n- [ ] Changes have been tested\n- [ ] New features include test cases\n- [ ] Documentation has been updated\n- [ ] Commit messages are clear\n- [ ] No unnecessary files added\n- [ ] Related issues are referenced\n\n### 4. Documentation Contributions\n\nHelp improve documentation:\n- Clarify existing content\n- Add examples\n- Fix typos\n- Add missing sections\n- Improve architecture diagrams\n\n## Branch Protection Rules\n\nThe main branch is protected. All changes require:\n- Pull request review\n- Passing tests/simulations\n- Clean commit history\n\n## Development Workflow\n\n1. **Fork** the repository\n2. **Create** a feature branch\n3. **Make** your changes\n4. **Test** thoroughly\n5. **Commit** with clear messages\n6. **Push** to your fork\n7. **Create** a Pull Request\n8. **Respond** to review comments\n9. **Merge** after approval\n\n## Tools & Environment\n\nRecommended tools for contribution:\n- **Simulation:** ModelSim, Questa, VCS\n- **Synthesis:** Vivado, ISE, Synopsys\n- **Linting:** Verialint, VerilogAnalyzer\n- **Editor:** VS Code (with Verilog extensions), Sublime Text\n\n## Getting Help\n\n- Check existing documentation in `/docs`\n- Review the README.md\n- Look at existing code examples\n- Open an issue with questions (tag as `question`)\n\n## License\n\nBy contributing, you agree your contributions will be licensed under the same MIT License as the project.\n\n## Recognition\n\nContributors will be recognized in:\n- GitHub contributors page\n- Release notes (for significant contributions)\n\nThank you for contributing! 🎉\n
